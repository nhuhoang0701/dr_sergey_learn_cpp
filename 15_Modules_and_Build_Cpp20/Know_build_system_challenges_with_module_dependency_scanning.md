# Know Build System Challenges with Module Dependency Scanning

**Category:** Modules & Build (C++20)  
**Standard:** C++20  
**Reference:** [P1689R5 — Format for describing dependencies of source files](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2022/p1689r5.html)  

---

## Topic Overview

C++20 modules fundamentally break the compilation model that build systems have relied on for decades. With headers, the build system knows all dependencies **before compilation** - they are explicit `#include` directives that can be extracted by simple text scanning or compiler-generated depfiles. With modules, **you cannot know what a file imports until you partially compile it**, because `import` declarations appear after preprocessor expansion and may be conditional.

This creates a **chicken-and-egg problem**: the build system needs to know the dependency graph to schedule compilation, but determining the dependency graph requires compilation (at least partial). The solution is a two-phase approach: **dependency scanning** (a fast pre-pass that extracts `import`/`export` declarations) followed by **DAG construction and ordered compilation**.

The reason this trips people up is that they expect modules to work the same way headers do - just faster. But the entire scheduling story changes. A build system that cannot do dependency scanning simply cannot support modules correctly.

```cpp
Traditional Headers:                    C++20 Modules:

  Source files ->> depfile scan         Source files ->> dependency scan (P1689)
       |              |                      |                |
       v              v                      v                v
  Known deps ->> parallel compile       Build DAG ->> ordered compilation
                                        (must compile providers before consumers)
```

The **P1689** format is the standardized JSON format for describing module dependencies. Compilers produce P1689 output during a scanning phase, and build systems consume it to construct the compilation DAG. This is the bridge between what the compiler knows and what the build system needs to schedule work correctly.

| Challenge | Description | Impact |
| --- | --- | --- |
| Dependency discovery | `import M;` requires knowing which file provides `M` | Build system must map module names -> files |
| Compilation ordering | BMI must exist before importers compile | Reduces build parallelism on DAG edges |
| Scanning overhead | Extra pre-compilation pass needed | Adds build latency (typically small) |
| Dynamic dependencies | Build DAG changes when module structure changes | Incremental build invalidation complexity |
| BMI portability | BMIs are compiler/flag-specific | No sharing across configurations |
| Circular imports | Modules cannot have circular dependencies | Must refactor code to break cycles |

The major build systems each handle this differently. CMake (3.28+) integrates dependency scanning into its generator pipeline, producing Ninja build files with dynamic dependencies. Ninja (1.11+) supports `dyndep` rules that allow the build graph to be augmented at build time. MSBuild handles scanning internally for MSVC. Meson, Bazel, and others are at varying levels of support.

## Self-Assessment

### Q1: What does P1689 dependency scanning output look like, and how does a build system use it

The compiler emits one P1689 JSON record per source file. Each record says what module (if any) the file provides, and which modules it requires. The build system reads all these records to build the DAG.

```json
// P1689 output for file: math.cppm
// Generated by: clang-scan-deps or cl.exe /scanDependencies
{
    "version": 1,
    "revision": 0,
    "rules": [
        {
            "primary-output": "math.cppm.o",
            "provides": [
                {
                    "logical-name": "math",
                    "source-path": "src/math.cppm",
                    "is-interface": true
                }
            ],
            "requires": [
                {
                    "logical-name": "utils",
                    "source-path": "src/utils.cppm"
                }
            ]
        }
    ]
}
```

And here is the record for a consumer that imports but does not provide any module.

```json
// P1689 output for file: main.cpp
{
    "version": 1,
    "revision": 0,
    "rules": [
        {
            "primary-output": "main.cpp.o",
            "requires": [
                {
                    "logical-name": "math"
                },
                {
                    "logical-name": "graphics"
                }
            ]
        }
    ]
}
```

Once the build system has all the P1689 records, it follows a fixed workflow to turn them into a build schedule.

**Build system workflow:**

```cpp
Step 1: Scan all source files -> collect P1689 JSON
Step 2: Build module-name -> source-file mapping:
          "math"     -> src/math.cppm
          "utils"    -> src/utils.cppm
          "graphics" -> src/graphics.cppm
Step 3: Construct DAG:
          utils.cppm  ->(provides utils)
          math.cppm   ->(requires utils, provides math)
          graphics.cppm ->(provides graphics)
          main.cpp    ->(requires math, graphics)
Step 4: Compile in topological order:
          utils.cppm    [first - no deps]
          math.cppm     [after utils]
          graphics.cppm [first - no deps, parallel with utils]
          main.cpp      [after math AND graphics]
```

### Q2: How does CMake handle module dependencies, and what are the practical pitfalls

CMake 3.28+ integrates P1689 scanning directly. You declare your module interface files with `FILE_SET CXX_MODULES`, and CMake takes care of scanning, DAG construction, and ordering. Here is a typical setup.

```cmake
# CMakeLists.txt - CMake 3.28+ module support
cmake_minimum_required(VERSION 3.28)
project(modular_app CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# CRITICAL: Use Ninja or Ninja Multi-Config generator
# cmake -G Ninja -B build

add_library(core_lib)
target_sources(core_lib
    PUBLIC
        FILE_SET CXX_MODULES FILES
            src/core.cppm              # export module core;
            src/core-types.cppm        # export module core:types;
    PRIVATE
        src/core_impl.cpp              # module core;
)

add_library(engine_lib)
target_sources(engine_lib
    PUBLIC
        FILE_SET CXX_MODULES FILES
            src/engine.cppm
)
target_link_libraries(engine_lib PUBLIC core_lib)

add_executable(app src/main.cpp)
target_link_libraries(app PRIVATE engine_lib)
```

The pitfalls table below is worth reading carefully before you hit these in practice - most of them produce confusing error messages.

**Pitfalls table:**

| Pitfall | Symptom | Fix |
| --- | --- | --- |
| Wrong generator | "CXX_MODULES not supported" | Use `-G Ninja` (not Makefiles) |
| CMake too old | `FILE_SET CXX_MODULES` not recognized | Upgrade to CMake 3.28+ |
| Missing `PUBLIC` on file set | Importers can't find BMI | Module files must be `PUBLIC` or `INTERFACE` |
| Ninja too old | `dyndep` errors | Upgrade to Ninja 1.11+ |
| Mixing compilers | BMI incompatibility | Use single compiler for entire module graph |
| Parallel build stalls | Build appears stuck | DAG serialization - reduce module depth |
| Partition not in same target | Linker errors | All partitions must be in the same CMake target |

With the targets set up correctly, CMake handles the rest automatically - `engine` waits for `core`, and `main.cpp` waits for `engine`.

```cpp
// src/engine.cppm
export module engine;
import core;                     // CMake resolves this because engine_lib links core_lib

export class Engine {
public:
    void run();
};

// src/main.cpp
import engine;                   // CMake handles DAG: core -> engine -> main
int main() {
    Engine e;
    e.run();
}
```

### Q3: How does the module dependency DAG affect build parallelism, and what strategies mitigate the impact

The DAG constraint is where module builds can actually feel slower than header builds despite the faster per-TU compilation. A deep chain forces serial compilation. A wide, flat structure lets most of the work run in parallel.

```cpp
Scenario: Project with deep module dependency chain

  Module DAG (deep):            Module DAG (wide, refactored):

  A -> B -> C -> D -> E            A --+-- B
  (serial: 5 steps)             |   +-- C
                                |   +-- D
                                |   +-- E
                                (parallel: 2 steps - A, then B|C|D|E)
```

Here is what the anti-pattern looks like in code, followed by the refactored version that allows parallel compilation.

```cpp
// ANTI-PATTERN: Deep chain - kills parallelism
// base.cppm
export module base;
export struct Config { /* ... */ };

// layer1.cppm
export module layer1;
import base;                     // depends on base
export class Logger { /* ... */ };

// layer2.cppm
export module layer2;
import layer1;                   // depends on layer1 (and transitively base)
export class Database { /* ... */ };

// layer3.cppm
export module layer3;
import layer2;                   // depends on layer2 -> layer1 -> base
export class Server { /* ... */ };

// BETTER: Flat structure with shared base
// base.cppm - same as above
export module base;
export struct Config { /* ... */ };

// logger.cppm - depends only on base
export module logger;
import base;
export class Logger { /* ... */ };

// database.cppm - depends only on base
export module database;
import base;
export class Database { /* ... */ };

// server.cppm - depends only on base
export module server;
import base;
export class Server { /* ... */ };

// app.cppm - fan-in at the top
export module app;
import logger;
import database;
import server;
```

The table makes the parallelism difference concrete.

**Parallelism analysis:**

| Structure | Modules | Critical Path | Max Parallelism |
| --- | --- | --- | --- |
| Deep chain (A->B->C->D->E) | 5 | 5 sequential compiles | 1 |
| Wide fan (A then B\|C\|D\|E) | 5 | 2 steps | 4 |
| Star (A, then all import A) | N+1 | 2 steps | N |
| Full module-per-class | Many | Depends on design | Varies |

**Strategies to maximize build parallelism:**

1. **Keep the module DAG shallow** - minimize transitive import chains.
2. **Use partitions** instead of separate modules for tightly coupled code - partitions within a module compile faster.
3. **Avoid "god modules"** - a single module that everything imports becomes a serialization bottleneck.
4. **Profile your dependency scan** - use `cmake --build build --verbose` to see compilation ordering.
5. **Consider header units** for widely-used, rarely-changing dependencies - their BMIs can be cached persistently.

## Notes

- **Dependency scanning is not optional** - build systems must perform it for any project using `import`. There is no way to manually declare module dependencies reliably (file contents determine them).
- **P1689** is the standard format. Compilers produce it via: `clang-scan-deps` (Clang), `cl.exe /scanDependencies` (MSVC), `g++ -fdep-file` (GCC, evolving).
- **Ninja's `dyndep`** is the runtime mechanism that injects scanned dependencies into the build graph. CMake generates the Ninja files that use this feature.
- **GNU Make does not support modules well** - it cannot handle dynamic dependency injection. Use Ninja or MSBuild.
- BMIs are **not portable** across compilers, compiler versions, or even different flag sets. CI systems must rebuild BMIs for each configuration.
- Module dependency DAGs must be **acyclic** - the compiler enforces this. If your current code has circular header includes, you must break the cycle before modularizing.
- **Scan time is fast** - typically 5-15% of full compilation time. The parallelism reduction from the DAG ordering is usually a bigger concern.
- Track your module DAG depth as a build-health metric. A critical path longer than 4-5 modules deep suggests architectural flattening is needed.
- `import std;` (C++23) collapses the entire standard library into one module, avoiding many small header-unit dependency edges.
