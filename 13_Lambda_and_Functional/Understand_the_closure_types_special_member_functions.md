# Understand the closure type's special member functions

**Category:** Lambda & Functional  
**Item:** #495  
**Standard:** C++11 (closures), C++20 (default-constructible/assignable stateless lambdas)  
**Reference:** <https://en.cppreference.com/w/cpp/language/lambda>  

---

## Topic Overview

Every lambda expression creates a **closure type** - a unique, unnamed class generated by the compiler. This class has an `operator()` and certain special member functions whose availability depends on the captures. Understanding which special members exist and when matters the moment you try to store a lambda in a container, copy it, or use it with `std::function`.

### Special Member Functions of Closures

The table summarizes what you get. Read it column by column: the captures you use determine what the closure type can and can't do.

| Member Function | Stateless `[]` | Value captures `[x]` | Reference captures `[&x]` | Move capture `[p = move(p)]` |
| --- | --- | --- | --- | --- |
| Default ctor | C++20: Yes | No | No | No |
| Copy ctor | Yes | Yes (if captures copyable) | Yes | No (unique_ptr) |
| Move ctor | Yes | Yes | Yes | Yes |
| Copy assign | C++20: Yes | No (deleted) | No | No |
| Destructor | Trivial | Yes | Trivial | Yes |
| Convert to fn ptr | Yes `+lambda` | No | No | No |

## Self-Assessment

### Q1: Explain which special members a lambda has: copy ctor if all captures are copyable, no default ctor

The best way to understand this is to ask the type system directly. `std::is_copy_constructible_v`, `std::is_default_constructible_v`, and friends will tell you exactly what the compiler generated for each closure type:

```cpp
#include <iostream>
#include <type_traits>

int main() {
    int x = 42;

    // Stateless lambda (no captures)
    auto stateless = [](int a) { return a * 2; };
    using S = decltype(stateless);
    std::cout << "Stateless lambda:\n";
    std::cout << "  default_constructible: " << std::is_default_constructible_v<S> << "\n";
    std::cout << "  copy_constructible:    " << std::is_copy_constructible_v<S> << "\n";
    std::cout << "  copy_assignable:       " << std::is_copy_assignable_v<S> << "\n";
    std::cout << "  trivially_copyable:    " << std::is_trivially_copyable_v<S> << "\n";

    // Lambda capturing by value
    auto by_val = [x](int a) { return a + x; };
    using V = decltype(by_val);
    std::cout << "\nBy-value capture lambda:\n";
    std::cout << "  default_constructible: " << std::is_default_constructible_v<V> << "\n";
    std::cout << "  copy_constructible:    " << std::is_copy_constructible_v<V> << "\n";
    std::cout << "  copy_assignable:       " << std::is_copy_assignable_v<V> << "\n";

    // Lambda capturing by reference
    auto by_ref = [&x](int a) { return a + x; };
    using R = decltype(by_ref);
    std::cout << "\nBy-reference capture lambda:\n";
    std::cout << "  default_constructible: " << std::is_default_constructible_v<R> << "\n";
    std::cout << "  copy_constructible:    " << std::is_copy_constructible_v<R> << "\n";
    std::cout << "  copy_assignable:       " << std::is_copy_assignable_v<R> << "\n";

    // Copy constructing a lambda - it copies the captured state
    auto copy1 = by_val;  // OK - int is copyable
    std::cout << "\nCopied by_val: " << copy1(10) << "\n";  // 52
}
// Expected output (C++20):
//   Stateless lambda:
//     default_constructible: 1
//     copy_constructible:    1
//     copy_assignable:       1
//     trivially_copyable:    1
//
//   By-value capture lambda:
//     default_constructible: 0
//     copy_constructible:    1
//     copy_assignable:       0
//
//   By-reference capture lambda:
//     default_constructible: 0
//     copy_constructible:    1
//     copy_assignable:       0
//
//   Copied by_val: 52
```

Notice that `by_ref` is still copy-constructible. The reference itself is copyable (it's just a pointer internally), so copying the closure copies the reference - you get two lambdas both referring to the same `x`. Copy assignment is deleted for all capturing lambdas, which is a deliberate design decision since assigning would be ambiguous semantically.

**Key rules:**

1. **No default ctor** (pre-C++20) - closure can only be created from the lambda expression
2. **Copy ctor** exists if ALL captured objects are copyable
3. **Copy assignment** is always **deleted** (even if captures are assignable) - until C++20 for stateless lambdas
4. **Move ctor** exists if ALL captured objects are movable

### Q2: Show that a lambda capturing a `unique_ptr` is move-only

When you move a `unique_ptr` into a lambda via init-capture, the closure type inherits that move-only nature. The compiler will tell you this quite clearly if you try to copy it:

```cpp
#include <iostream>
#include <memory>
#include <type_traits>
#include <vector>

int main() {
    auto ptr = std::make_unique<int>(100);

    // Move unique_ptr into lambda via init-capture
    auto lambda = [p = std::move(ptr)]() {
        std::cout << "Value: " << *p << "\n";
    };

    using L = decltype(lambda);
    std::cout << std::boolalpha;
    std::cout << "copy_constructible: " << std::is_copy_constructible_v<L> << "\n";
    std::cout << "move_constructible: " << std::is_move_constructible_v<L> << "\n";
    std::cout << "copy_assignable:    " << std::is_copy_assignable_v<L> << "\n";

    // auto copy = lambda;           // ERROR: deleted copy constructor!
    auto moved = std::move(lambda);  // OK: move constructor works
    moved();

    // Practical impact: can't store in std::function (requires copyable!)
    // std::function<void()> f = std::move(moved);  // ERROR in most implementations

    // Workaround: use std::move_only_function (C++23)
    // std::move_only_function<void()> f = std::move(moved);  // OK!

    // Or wrap in shared_ptr for std::function compatibility:
    auto shared = std::make_shared<decltype(moved)>(std::move(moved));
    std::function<void()> f = [shared]() { (*shared)(); };
    f();
}
// Expected output:
//   copy_constructible: false
//   move_constructible: true
//   copy_assignable:    false
//   Value: 100
//   Value: 100
```

The `std::function` incompatibility is a practical problem that comes up frequently. `std::function` requires the stored callable to be copyable (it copies it internally), so a move-only lambda simply won't fit. The `shared_ptr` workaround shown here is a common solution, though C++23's `std::move_only_function` is the clean answer.

### Q3: Demonstrate that C++20 allows default construction and copy assignment of capture-less lambdas

This is a genuinely useful C++20 improvement. Before C++20, using a lambda as a container comparator required passing the lambda instance to the container constructor. In C++20, the stateless closure type is default-constructible, so the container can construct the comparator itself:

```cpp
#include <iostream>
#include <set>
#include <map>
#include <type_traits>

int main() {
    // C++20: stateless lambdas are default-constructible
    auto cmp = [](int a, int b) { return a > b; };
    using Cmp = decltype(cmp);

    std::cout << std::boolalpha;
    std::cout << "default_constructible: " << std::is_default_constructible_v<Cmp> << "\n";
    std::cout << "copy_assignable:       " << std::is_copy_assignable_v<Cmp> << "\n";

    // Default-construct the lambda type - no instance needed!
    Cmp cmp2{};  // C++20: OK! Before C++20: ERROR
    std::cout << "cmp2(1, 2): " << cmp2(1, 2) << "\n";  // false (1 > 2 is false)

    // Practical benefit: use lambda type as container comparator
    // Before C++20, std::set<int, decltype(cmp)> required passing cmp to constructor.
    // C++20: the set can default-construct the comparator itself!
    std::set<int, Cmp> s;  // C++20: works without passing cmp instance
    s.insert(3);
    s.insert(1);
    s.insert(5);
    s.insert(2);

    std::cout << "Sorted descending: ";
    for (int x : s) std::cout << x << " ";
    std::cout << "\n";

    // Copy assignment (C++20)
    auto f1 = [](int x) { return x + 1; };
    auto f2 = [](int x) { return x + 1; };
    // f1 and f2 are DIFFERENT types (each lambda has unique type)
    // f1 = f2;  // ERROR: different types!

    // But same-type assignment works in C++20:
    decltype(f1) f3 = f1;  // copy construct
    f3 = f1;               // C++20: copy assign - OK!
    std::cout << "f3(10): " << f3(10) << "\n";

    // Conversion to function pointer
    int (*fp)(int, int) = cmp;  // stateless -> function pointer
    std::cout << "fp(5, 3): " << fp(5, 3) << "\n";   // true (5 > 3)

    // Also via unary +
    auto raw_fp = +cmp;  // decays to function pointer
    std::cout << "raw_fp(1, 1): " << raw_fp(1, 1) << "\n";  // false
}
// Expected output:
//   default_constructible: true
//   copy_assignable:       true
//   cmp2(1, 2): false
//   Sorted descending: 5 3 2 1
//   f3(10): 11
//   fp(5, 3): true
//   raw_fp(1, 1): false
```

The `+cmp` trick at the end is worth knowing: the unary `+` forces the lambda to decay to a function pointer. It's a neat way to get a raw pointer without writing out the full function pointer type.

## Notes

- **Each lambda has a unique type.** Even `auto f = [](int x){ return x; }; auto g = [](int x){ return x; };` - `decltype(f)` is not the same type as `decltype(g)`. They just happen to behave identically.
- **`+lambda` trick:** The unary `+` forces conversion of a stateless lambda to a function pointer.
- **`std::function` requires copyability.** Move-only lambdas need C++23 `std::move_only_function` or a custom wrapper.
- **Mutable lambdas:** Adding `mutable` doesn't affect which special members exist - it only makes `operator()` non-const, allowing modification of by-value captures.
- **Closure size:** `sizeof(closure)` = sum of captured members (with padding). Empty closure (no captures) has `sizeof == 1` (EBO may reduce to 0 inside containers).
- **C++23 update:** `static operator()` for lambdas - even less overhead for stateless lambdas.
